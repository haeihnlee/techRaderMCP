# How to design a multi-agent system that skips the LLM

- **컨퍼런스**: Google Cloud Next
- **출처**: https://www.youtube.com/watch?v=Fzd0BWMH65s
- **요약 일시**: 2026-06-07 09:03:56

---

## 🔑 핵심 요약
- **1,000개의 AI 에이전트**가 동시에 마라톤을 뛰는 오픈소스 시뮬레이션 시스템 — 핵심은 *"경주가 시작되면 대부분의 에이전트는 LLM을 거의 쓰지 않는다"*
- **"언제 LLM에게 결정을 맡기고, 언제 코드로 처리할 것인가"** 라는 멀티에이전트 아키텍처의 핵심 의사결정을 마라톤 경로 설계·시뮬레이터 두 사례로 설명
- **결정론적(deterministic) 로직**으로 처리하되, *어떤 알고리즘을 쓸지 고르는 단계*에서만 `Gemini`를 활용 — 런타임이 아닌 설계 타임에 AI 사용

---

## 📣 주요 발표 내용
- 마라톤 경로 설계: 플래너 에이전트가 `load skill` 툴로 **GIS 공간 엔지니어링** 스킬을 로드해 경로 생성
- 경로는 **정확히 26.2188 마일**, 시(市) 승인 도로망(`GeoJSON` 표준 파일) 위에서만 가능하도록 제약
- `Google Maps`를 못 쓴 이유:
  - 운전 경로는 **일방통행**을 따르지만 마라톤은 도로를 막고 양방향 주행 가능
  - 도보 경로는 라스베이거스에서 **육교·건물 통과**(예: Luxor 호텔 관통) 경로를 자주 생성
- 경로 탐색은 **NP-hard 문제** → LLM은 제약이 늘수록 노이즈가 커지고 컨텍스트 유지 실패
- 채택한 **결정론적 알고리즘 (Spine & Sprout)**:

| 단계 | 알고리즘 / 기법 | 역할 |
|------|----------------|------|
| Phase 1 | `Dijkstra` + **Haversine 가중 엣지** | 도로 그래프로 랜드마크 연결 |
| Phase 2 | **Serpentine waypoint** | 경로 연장 후 마지막 구간 트림해 정확한 거리 맞춤 |

- 알고리즘 *선정*에는 `AI Studio`의 **code execution** + **Google Search 그라운딩** 활용 → Gemini가 알고리즘을 검색·Python 코드 생성
- 이후 `Gemini CLI`로 가져와 *"unit test 가능하게, 딥 코드리뷰"* 요구하며 수차례 반복 개선

## 💡 개발자 포인트
- **시뮬레이터 에이전트**도 동일한 원칙 적용 — 에이전트 프레임워크(생명주기·`A2A` 통신·텔레메트리)는 유지하되, 기계적 작업에는 LLM 호출을 넣지 않음
- 게임 개발의 **server-side tick** 개념 차용: 중앙 서버가 매 틱마다 전체 러너의 상태·속성을 추적 (시뮬레이터=게임 서버, 러너=플레이어)

> **핵심 트릭 (ADK `before_model_callback`)**
> `pre-race agent`와 `tick agent`는 `LlmAgent`라 model 필드가 *필수*지만, **실제로는 model이 한 번도 호출되지 않음**.
> `before_model_callback`이 모든 invocation을 가로채 **결정론적 tool call을 직접 반환**하기 때문 (agent.py 252번째 줄 주석 참고).
> → 에이전트 패러다임은 유지하면서 LLM 비용·지연을 0으로 만드는 패턴.

- **LLM을 써야 할 때 vs. 결정론 로직을 써야 할 때**:
  - 다양한 사용자 입력의 edge case를 모두 처리할 수 없을 때 → **LLM 결정**이 유리
  - 특정 use case(예: 정해진 경로 플래닝)는 → **결정론 로직으로 LLM 생성을 우회(bypass)**
- 실무 팁: `before tool call` 콜백으로 특정 동작을 가로채 백엔드에서 결정론 로직을 강제 실행 가능
- 핵심 교훈: **"모델이 도움이 되는 곳에만 쓰고, 도움이 안 되는 곳에선 건너뛴다"** — 1,000 에이전트 운영을 *경제적으로* 만든 결정
