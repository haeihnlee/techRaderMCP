## 🔑 핵심 요약
- **PayPal**이 인시던트를 **감지→분류(triage)→완화→보고**까지 처리하는 **자율 에이전트 생태계**를 프로덕션에 구축한 사례
- **Gemini Enterprise + ADK** 도입으로 세션/메모리 관리·병렬 실행 등을 직접 구현할 필요가 없어져 **코딩량 50~60% 감소**
- 에이전트 데이터 접근은 **MCP 서버**를 통한 통합 레이어로 제공 — 과거 인시던트 이력 등 전사 데이터 활용
- 프로덕션 신뢰 확보의 핵심은 **내장 옵저버빌리티**(실행 경로·추론 과정·모델 평가 추적)

---

## 📣 주요 발표 내용
- 단일 에이전트가 아닌 **에이전트 생태계**: `supervisor agent`, `reporting agent`, `change analyst agent`, `service agent`가 협업하여 인시던트를 분류하고 해결책 도출
- **Gemini Enterprise** 프레임워크가 기본 제공하는 기능: **세션 관리**, **메모리 관리**, **영속성(persistence)**, **에이전트 병렬 실행**
- 옵저버빌리티가 **out-of-the-box** 제공: 실행 경로, 각 단계의 추론 근거, 모델 정확도·할루시네이션 여부 평가
- **모델 스왑 지원**: 에이전트 코드를 변경하지 않고 하위 모델만 교체 가능 → 신모델 출시 시 QA에서 빠르게 성능 검증
- 지식베이스·SOP 등을 담는 **영속 스토리지**와 인간의 지속적 피드백으로 에이전트가 점점 개선됨 (에이전트 = "복귀한 인턴" 비유: 컨텍스트는 있으나 감독 필요)
- PayPal 전사 데이터는 **MCP 서버** 기반 통합 레이어로 에이전트에 노출

---

## 💡 개발자 포인트
- 자율 에이전트 도입은 **read-only 에이전트부터 시작**하는 것이 정석 — 동작을 관찰하며 신뢰를 쌓은 뒤 확장

> ⚠️ **프로덕션을 변경(mutate)하는 에이전트는 가장 위험**하다. 반드시 **allow list / block list**를 적용하고, **컨테이너·샌드박스**에 격리하며, **supervised mode**로 운영해야 한다.

- 에이전트 성공의 전제 조건은 **데이터 인프라**: 에이전트가 독립적으로 조회할 수 있는 통합 데이터 레이어(MCP 서버 등)가 먼저 갖춰져야 함
- 실행 경로·추론 추적이 가능한 **옵저버빌리티**가 없으면 프로덕션 전환 신뢰 확보가 어려움 — 프레임워크 선택 시 우선 확인할 것
- 오케스트레이션·메모리·병렬 실행을 직접 구현하기보다 프레임워크 제공 기능을 활용하면 개발 속도가 크게 향상
