# Real time fraud detection with AlloyDB AI

- **출처**: https://www.youtube.com/watch?v=aYCuGy97Nf4
- **요약 일시**: 2026-05-19 09:07:45

## 🔑 핵심 요약
- **AlloyDB AI**의 벡터 임베딩 + `ScaNN` 인덱스로 **실시간 사기 탐지 엔진** 구축 사례 시연
- 벡터 유사도만으로는 **재현율(recall)과 오탐(false positive)의 트레이드오프** 발생 → **Gemini 추론**을 결합한 하이브리드 접근으로 해결
- 신규 **`AI.IF` 함수**와 in-database 실행으로 **초당 100,000행 처리**, 추론 비용을 **약 6,000배 절감**(거래당 1/10 센트)

---

## 📣 주요 발표 내용
- **시나리오**: 가상 금융사 *Cymbal Investments*가 레거시 Oracle/SQL Server를 **AlloyDB**로 마이그레이션, 200만+ 거래 데이터 활용
- **임베딩 생성**: 거래 테이블 데이터를 텍스트화한 feature string에 `embedding()` 함수 적용 → 수백만 벡터를 수 분 내 생성
- **벡터 인덱스 `ScaNN`**
  - Google의 14년 연구 기반 최신 알고리즘
  - **HNSW 대비 메모리 사용량 최대 4배 감소**
  - **100억+ 벡터 스케일** 최적화, 고속 금융 스트림에 적합
- **하이브리드 탐지 로직**
  - 알려진 사기 거래와의 **벡터 거리(nearest neighbor)** 로 1차 분류
  - 임계값 근처 거래는 `AI.IF()` 로 **Gemini 자연어 추론** 호출 (예: "이 사용자의 평소 소비 패턴 대비 카드 테스트 시도인가?")
  - 재현율 +5%, 오탐 -5% 개선
- **성능 최적화**
  - **Array 기반 처리**: 초당 7,700행 (행 단위 호출 대비 2,000배)
  - **최적화된 AI 함수** + native in-database 실행 + surrogate model: **초당 100,000행** (행 단위 대비 약 23,000배)

## 💡 개발자 포인트
- **DB 안에서 LLM 추론**을 직접 호출 가능 → 별도 추론 서버/파이프라인 없이 트랜잭션 흐름에서 즉시 의사결정
- 벡터 검색을 **검색 외 용도**(중복 제거, 비주얼 서치, 엔티티 레졸루션, 이상 탐지)로 적극 확장 가능
- 임계값 튜닝 예시 (재현율 vs 오탐)

  | 임계값 | 재현율(Recall) | 비고 |
  |--------|----------------|------|
  | 0.031  | ~75% 미만      | 사기 1/4 누락 |
  | 0.021  | 79%            | 기본 시작점 |
  | 0.011  | 93%            | 정상 거래 약 20% 오탐 |

> **트레이드오프 해소 패턴**: 벡터 유사도가 임계값 근처인 "회색지대" 거래에만 `AI.IF`로 LLM 호출 → 전체 성능을 해치지 않으면서 정확도 확보

> **비용 주의**: 거래마다 LLM 호출 시 DB 성능·비용이 폭증할 수 있음. AlloyDB의 **batched / native AI function**을 쓰지 않으면 동일한 효과를 기대하기 어려움

- 활용 가능 함수: `embedding()`, `AI.IF()`, **`ScaNN` 벡터 인덱스**
