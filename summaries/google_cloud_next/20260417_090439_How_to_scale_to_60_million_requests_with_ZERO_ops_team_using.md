# How to scale to 60 million requests (with ZERO ops team) using Cloud Run

- **컨퍼런스**: Google Cloud Next
- **출처**: https://www.youtube.com/watch?v=mpgoqDUAN6k
- **요약 일시**: 2026-04-17 09:04:39

---

## 🔑 핵심 요약

- **1인 팀**으로 월 **6,000만 요청**을 처리하는 피처 플래그 서비스 `Rocket Flag`를 **Cloud Run** 만으로 운영
- **서버리스 아키텍처** 채택으로 SRE에 소요되는 시간 **0분**, 월 운영비 약 **USD $180** 달성
- DB 비용 폭증 문제를 **인메모리 배치 처리** 시스템으로 해결하여 `Firestore` 쓰기 비용 대폭 절감

---

## 📣 주요 발표 내용

### 🏗️ 아키텍처 구성

- 🌍 **멀티 리전** 배포: `Cloud Run` 서비스를 여러 리전에 독립 배포 (현재 Western Europe이 가장 트래픽 많음, ~26 req/s)
- ⚡ 애플리케이션 언어로 **Go** 선택 → 짧은 **콜드 스타트** 시간 및 고성능 처리
- 🐳 **멀티 스테이지 Docker 빌드** 적용 → 최종 바이너리만 `scratch` 컨테이너에 복사하여 보안 강화
- 🗄️ 데이터 저장소 이중화:
  - `Firestore` → 애플리케이션 상태 (피처 플래그 카운터)
  - `BigQuery` → 분석용 이벤트 적재

### 🛡️ 보안 강화

- **Cloud Armor** 를 애플리케이션 앞단에 배치
- 유효한 URL 패턴을 **정규식(RegEx)** 으로 정의, 미매칭 요청은 `Cloud Run`에 도달하기 전에 차단
- 불필요한 로그 노이즈 제거 및 악성 트래픽 필터링 효과

### 💰 비용 최적화 (배치 처리 도입)

- **문제**: 요청마다 `Firestore`에 카운터 증가 + `BigQuery`에 레코드 적재 → 쓰기 비용 급증
- **해결**: **Go 루틴(goroutine)** 을 활용한 인메모리 배치 처리 시스템 구축
  - 요청 수신 시 카운터를 **메모리에 누적**
  - **1분에 1회** `Firestore`에 단건 배치 쓰기, `BigQuery`에 배치 로드 잡 실행
- 대안 패턴: 트래픽이 낮은 경우 **`Pub/Sub` 큐** → 워커가 소비 후 배치 DB 쓰기 권장

## 💡 개발자 포인트

### ✅ 서버리스 선택 기준

- 복잡한 인프라 없이 **멀티 리전 고가용성** 확보 가능
- **자동 스케일링**으로 트래픽 급증 대응 (2M → 60M req/month 자연스럽게 성장)
- VM 기반 대비 **총 소유 비용(TCO)** 절감 (특히 인건비·운영 비용 포함 시)

### ⚠️ 서버리스 DB 사용 시 주의사항

> **`Firestore`, `BigQuery` 등 서버리스 DB는 쓰기 단위 과금**이므로, 요청당 직접 쓰기 패턴은 트래픽 증가 시 **비용 폭탄**으로 이어질 수 있음. 반드시 **배치 처리 전략**을 초기 설계부터 고려할 것.

### ⚠️ Cloud Run CPU 활성화 조건 주의

> `Cloud Run`은 **요청 처리 중에만 CPU가 활성화**됨. 따라서 Go 루틴 기반 백그라운드 배치는 **트래픽이 충분히 높을 때**만 안정적으로 동작함. 트래픽이 낮거나 간헐적인 서비스라면 **`Pub/Sub` + 별도 워커** 패턴을 사용할 것.

### 🚩 피처 플래그 vs Cloud Run 트래픽 스플리팅 비교

| 항목 | `Cloud Run` 트래픽 스플리팅 | 피처 플래그 (`Rocket Flag`) |
|---|---|---|
| 배포 단위 | 컨테이너 버전 | 코드 내 기능 단위 |
| 롤백 방법 | 재배포 필요 | 버튼 클릭 즉시 롤백 |
| 타겟팅 | 전체 트래픽 비율 | 특정 사용자 그룹 지정 가능 |
| 비기술직 사용 | Google Cloud Console 접근 필요 | 별도 웹 UI 제공 |
| 적용 범위 | Cloud Run 한정 | 플랫폼 무관 |
