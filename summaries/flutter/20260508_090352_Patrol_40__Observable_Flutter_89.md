## 🔑 핵심 요약
- **Patrol 4.0** 출시 — LeanCode가 만든 Flutter용 **네이티브 E2E 테스트 프레임워크**의 사실상 첫 "정식 1.0" 릴리스
- 가장 큰 변화는 **웹(Web) 지원 추가**. GitHub 이슈 1순위였던 기능으로, 이제 모바일·데스크톱·웹을 한 프레임워크로 커버
- 웹 지원 구현은 **AI 바이브 코딩(vibe coding)** 으로 주말에 PoC가 완성되어, 두려워했던 대형 작업이 의외로 빠르게 풀린 사례

---

## 📣 주요 발표 내용
- **Patrol의 지향점**: 외부 farm(BrowserStack, Firebase Test Lab, Sauce Labs 등)에서 돌릴 때 **"Flutter라는 사실을 숨기고"** Android/iOS 네이티브 테스트처럼 보이도록 설계 — 엔터프라이즈 QA 부서가 기존 farm 계약 그대로 사용 가능
- **`patrol` 4.0의 웹 지원 핵심 결정**
  - 웹 E2E 프레임워크로 **`Playwright`** 채택 (Puppeteer, Selenium 대안 중 가장 트렌딩·표준화 진행)
  - Gemini와의 비교 분석 후 선정
- **모바일 vs 웹 아키텍처 차이점**
  - 모바일: "test app"이 메인 앱을 구동하는 구조
  - 웹: 그런 호스트 앱 개념이 없음 → URL을 브라우저로 띄우는 방식이라 **discovery phase**(Dart 런타임에 테스트 개수 묻기) 재설계 필요
- **macOS 지원**도 함께 정착 — iOS와 API가 거의 같아 비교적 빠르게 안착

## 💡 개발자 포인트
- Flutter 통합 테스트의 한계(네이티브 영역과 상호작용 불가)를 **`patrol`** 이 메워주므로, **엔터프라이즈 QA 정책상 Flutter 도입이 막혔던 팀**에 좋은 돌파구
- 기존 모바일 farm 계약을 그대로 쓰면서 Flutter 앱을 검증할 수 있음 → **CI 파이프라인 재구축 비용 절감**
- 웹 지원이 추가되면서 **Android / iOS / macOS / Web** 단일 테스트 프레임워크로 통일 가능

> **개발 의사결정 인사이트**: "이 기능은 6개월짜리야"라며 미뤄둔 작업이 실제로는 며칠이면 끝나는 경우가 많다. **AI 바이브 코딩으로 PoC를 먼저 만들어보면**, "얼마나 큰가?"라는 질문이 "프로덕션 레디까지 얼마나 남았나?"로 바뀌어 견적이 훨씬 명확해진다.
