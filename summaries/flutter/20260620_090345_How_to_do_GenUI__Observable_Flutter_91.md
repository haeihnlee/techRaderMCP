## 🔑 핵심 요약
- **GenUI(Generative UI)**는 LLM에게 UI 코드를 짜게 하는 게 아니라, **LLM이 구조화된 JSON 출력**을 내면 이를 **내 위젯 카탈로그에 매핑**해 화면을 그리는 개념이다.
- Flutter GDE Chowtoy가 직접 만든 **핀란드어 학습 앱(언어 학습이 아닌 "정착 통합" 앱)** 에 GenUI와 **Firebase AI + Gemini Nano Banana Pro**를 적용한 사례를 공유.
- 과거 **STT(speech-to-text)** 전용 모델 방식에서, 이제는 **Gemini가 음성 분석·문법 교정·채점**까지 한 번에 처리하는 멀티모달 방식으로 진화.

---

## 📣 주요 발표 내용
- **GenUI가 아닌 것**: LLM이 직접 Flutter UI 코드를 작성하는 것 → 흔한 오해.
- **GenUI의 핵심 동작 흐름**
  - 개발자가 **유스케이스 + 위젯 카탈로그**를 LLM(Gemini)에 제공
  - LLM이 "이 컴포넌트들을 이렇게 조합하라"는 **위젯 트리(구조화 출력)** 를 반환
  - **`surface` 위젯**이 그 위젯 트리를 받아 실제 화면에 렌더링
  - 즉 LLM에게 **캔버스를 주고, 무엇을 그릴지 지시받는** 구조.
- **이미지 기반 언어 학습 (블로그 시리즈 1편)**
  - **`Nano Banana Pro`(Gemini 이미지 모델)** 로 학습용 이미지를 생성
  - 사용자가 이미지를 보며 주변 사물 어휘를 말하거나 작성 → **시각적 학습**으로 뇌의 인지 강화
  - 독일어 시험의 "이미지 설명" 섹션에서 착안한 방식.
- **음성 평가 파이프라인**
  - 사용자가 말하면 **Gemini가 오디오를 분석 → 실수 교정 → 채점**
  - 프롬프트에 **공식 평가 기준**을 첨부하니 Gemini가 기대 이상으로 정확하게 채점.

## 💡 개발자 포인트
> **GenUI ≠ "LLM이 UI 코드를 생성"**. LLM은 텍스트(JSON)만 내고, 실제 렌더링은 내가 정의한 위젯 카탈로그가 담당한다는 점이 핵심 — 안정성·디자인 일관성을 유지하면서 동적 화면 구성이 가능하다.

- **위젯 카탈로그 설계가 GenUI의 품질을 좌우** — LLM이 조합할 수 있는 재료를 미리 잘 정의해야 한다.
- **`surface` 위젯** 패턴: LLM 출력(위젯 트리)을 받아 화면에 주입하는 렌더링 진입점 역할.
- **Firebase AI + Gemini** 조합으로 모바일 앱에서 멀티모달(이미지 생성·오디오 분석)을 직접 호출 가능.
- 평가/채점 같은 도메인 로직은 **프롬프트에 공식 기준(rubric)을 명시**하면 LLM 정확도가 크게 올라간다.
- 단일 언어만 지원하는 이유는 기술적 한계가 아니라 **콘텐츠 제작 비용** — 신규 언어 추가는 코드보다 콘텐츠가 병목.
