## 🔑 핵심 요약
- AI 에이전트 컨텍스트 제공 방식이 **System Instructions → Rules File → MCP Server → Skills** 순으로 진화해왔음
- **Skills**는 마크다운 + YAML 프론트매터 기반으로, 컨텍스트를 **지연 로딩(Lazy Loading)** 하여 비용·속도 측면에서 효율적
- **MCP Server**는 "유틸리티(도구 실행)" 담당, **Skills**는 "태스크/워크플로우 가이드" 담당으로 역할이 구분됨

---

## 📣 주요 발표 내용

### AI 컨텍스트 제공 방식의 진화

- 📌 **System Instructions**: 채팅 시작 시 모델 동작 방향 설정 (예: "당신은 Flutter 전문가입니다")
- 📄 **Rules File**: 에이전트 하니스가 파일을 읽어 채팅 시작 시 컨텍스트로 주입 → 업데이트 시 새 채팅 필요
- 🔧 **MCP Server**: 명확하고 범위가 좁은 도구 실행에 최적 (예: `dart format`, `pub search`, 파일 분석)
  - 단점: MCP 서버가 쌓이면 **150개 이상의 툴 정의**가 매 요청마다 컨텍스트에 포함되어 비용 폭증
- ⭐ **Skills (현재 표준)**: 위 문제를 해결한 최신 방식

### Skills의 핵심 특징

- 🗂️ 구조: **마크다운 파일** + 상단 `YAML` 프론트매터(frontmatter)
- ⚡ **지연 로딩**: 초기에는 스킬의 **이름과 설명만** 모델에 전달 (300개 스킬이 있어도 300개의 제목/설명만 전송)
- 🧠 모델이 필요하다고 판단할 때만 해당 스킬의 **전체 내용을 추가 로딩**
- 🔗 스킬 내부에서 MCP 도구 호출, 외부 리소스 페치, 스크립트 실행 등을 안내 가능

## 💡 개발자 포인트

### MCP Server vs Skills 역할 구분

| 구분 | MCP Server | Skills |
|------|-----------|--------|
| 역할 | 유틸리티 (도구 실행) | 태스크/워크플로우 가이드 |
| 예시 | `dart format`, `dart analyze`, pub 검색 | "Flutter 앱에 FFI 추가하는 방법" |
| 컨텍스트 비용 | 항상 전체 툴 정의 포함 | 이름+설명만 → 필요 시 전체 로딩 |
| 로딩 방식 | 즉시(Eager) | 지연(Lazy) |

### 실전 활용 패턴

- 🛒 **고객 지원 봇 예시**:
  - MCP: `refundOrder()` 도구 실행
  - Skill: 환불 요청 고객 응대 방법, 제공 가능한 옵션, 커뮤니케이션 톤 가이드
- 🔄 **반복 워크플로우 자동화**: 매번 같은 작업 패턴을 Skills로 정의하면 모델이 일관되게 수행
- 🧩 **RAG와의 조합**: 쿼리 전에 관련 스킬 목록을 RAG로 필터링 후 모델에 전달 가능

> ⚠️ **주의**: MCP 서버를 여러 개 설치하면 첫 메시지 입력 전부터 컨텍스트 윈도우가 상당히 소모됨. 반복적·복잡한 워크플로우는 Skills로 분리하는 것을 권장.

> 💡 **핵심 인사이트**: Skills는 단순한 임시방편(band-aid)이 아니라, **소형 모델에서도 복잡한 태스크를 빠르고 저렴하게** 수행할 수 있게 해주는 구조적 접근법.
